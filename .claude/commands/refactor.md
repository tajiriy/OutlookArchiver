name: refactor
description: リファクタリングトラッカーから次の項目を選んで実行する
---

docs/refactoring_tracker.md を読み込み、次に対応すべきリファクタリング項目を1つ選んで実行してください。

## 項目のソース

tracker の項目は以下のソースから登録される:
- **review**: code-reviewer サブエージェントのレビュー指摘から（ユーザー承認後に登録）
- **manual**: ユーザーが手動で追加
- **implementation**: 実装中に発見

## 手順

1. **対象選択**: refactoring_tracker.md から `open` ステータスの項目を優先度順（Critical → High → Medium → Low）に確認し、次に着手すべき1項目を選ぶ。関連性の高い項目は同時対応を検討する。

2. **現状調査**: 対象の関連ファイルを読み込み、リファクタリングの影響範囲を把握する。

3. **実装**: 対策欄の方針に従い、最小限の変更でリファクタリングを実施する。
   - 既存の動作を変えない（振る舞い保持リファクタリング）
   - 過剰な修正を避ける（対象項目のスコープ内に留める）

4. **ビルド・テスト確認**:
   - `./build.sh` でビルドエラーがないことを確認する
   - `./test.sh` でテストが通ることを確認する

5. **ドキュメント更新**:
   - refactoring_tracker.md: 対象項目のステータスを `done` に、修正日を記入、対策欄に実施内容を追記、サマリテーブルのカウントを更新、変更履歴に追記する
   - CLAUDE.md: ディレクトリ構成やファイル説明に変更があれば更新する
   - 他の項目への影響（obsolete になるものなど）があればそれも更新する

6. **完了報告**: 以下の構成で報告する。コミットは行わない（/ship で別途指示する）。
   - **問題**: 何が問題だったのか（tracker の「内容」を要約）
   - **対応**: どう修正したのか（具体的な変更内容）
   - **変更ファイル一覧**: 各ファイルの変更内容（新規/修正/削除行数）
   - **ビルド結果**: `msbuild` の結果

7. **手動テストガイド**: 変更内容に応じて、手動で確認すべきポイントを簡潔に提示する。

## 注意事項

- 1回の実行で対応する項目は1〜2件（関連する場合のみ2件）に留める
- 対象項目が他の未完了項目に依存する場合は、先に依存先を対応する
- 実装中に新たなリファクタリング対象を発見した場合は refactoring_tracker.md に追記する（ステータス `open`）
- wontfix にすべきと判断した場合は理由をメモに記載してステータスを変更する

## refactor-worker（worktree）を使った並列実行時の注意

複数項目を並列実行する場合は以下のルールに従う：

### グループ化
- **同一ファイルを変更する項目は必ず1つの worker にまとめる**（別々の worker が同一ファイルを変更すると、マージ時に一方の変更が失われる）

### worker への指示
- refactor-worker には**必ずコミットさせる**（「コミット不要」と指示しない）
- コミットしないと worktree ブランチにポインタが残らず、変更を取り込めない

### マージ手順（重要）
worktree の結果をメインに取り込む際、`git checkout <branch> -- <file>` は**使わない**。ファイル全体を上書きするため、メインで既にコミット済みの変更が失われる。

代わりに以下の手順を使う：
1. `git -C <worktree_path> diff main -- <file>` で worktree の差分を確認する
2. 差分の内容を Edit ツールでメインのファイルに手動適用する
3. または `git merge <worktree-branch>` でマージする（worktree 側がコミット済みの場合）

### マージ後の検証
- マージ後に必ず全項目の反映を検証する（Explore エージェントで突き合わせ）
- ビルド・テストを実行して問題がないことを確認する
